Kubernetes vs Docker: From a Single Box to a Fleet

A continuation of the Docker internals series — what happens when one container host is no longer enough, and what Kubernetes actually does under the hood
Technology May 5, 2026 12 min read

[Disclaimer: This blog was written solely for my understanding purpose only. Any mistakes found that need to be addressed, please feel free to reach out to me.]

Where the Previous Posts Left Off

In the first post of this series, I covered what a container actually is — namespaces, cgroups, and Union File Systems. In the second post, I covered how containers talk to each other — default bridges, user-defined networks, veth pairs, iptables, and how Docker Compose automates all the wiring.

Both of those posts lived in a world with one assumption baked in: one machine. Everything ran on a single host. That assumption is fine for development. It breaks down the moment you need to run your app reliably at scale.

That's where Kubernetes enters.

1. The Problem Docker Compose Can't Solve

Imagine your web app is running as three Docker containers behind Compose on a single server. Traffic spikes. The server's CPU hits 100%. You have two options: manually SSH in and restart things, or spin up another server and figure out how to spread load across both. Neither is acceptable for a production service that needs to be up 24/7.

Compose is a single-host tool. It has no concept of multiple servers, no ability to move containers between machines, and no understanding of whether a container is actually healthy or just running. If a container crashes, Compose can restart it — but only if you're still on the same machine. If the machine itself goes down, your app is gone.

This is the gap Kubernetes fills.

2. Docker Creates Containers. Kubernetes Manages Them.

This is the single most important distinction to internalize. Docker and Kubernetes are not competing products — they are tools for different levels of the same stack.

  • Docker answers: "How do I package and run a single application in an isolated environment?"
  • Kubernetes answers: "How do I run thousands of containers, across dozens of servers, reliably, with automatic failover and scaling?"

A good analogy: if you are moving houses, Docker is the standardised cardboard box — it packages your things so they survive transport identically. Kubernetes is the logistics manager and crane operator at a shipping port — it decides which ship each box goes on, reroutes boxes when a ship is delayed, and makes sure nothing gets lost.

Feature Docker (+ Compose) Kubernetes
Primary goal Run containers on one host Orchestrate containers across a cluster
Scope Single machine Many machines (nodes)
Scaling Manual Automatic (horizontal auto-scaling)
Self-healing Restart on same host only Restart, reschedule to healthy nodes
Load balancing Not built in Built in, across all nodes
Rolling updates Manual Zero-downtime by default
Complexity Low — easy for developers High — requires operational expertise

3. How They Work Together in Practice

In a 2026 production workflow, Docker and Kubernetes are almost always used together in sequence:

Developer writes code
    │
    ▼
docker build → creates a container image
    │
    ▼
docker push → uploads image to a registry (Docker Hub, ACR, ECR)
    │
    ▼
Kubernetes pulls the image
    │
    ▼
Kubernetes decides which node has capacity
    │
    ▼
Container starts running on that node

Docker handles the "what" — it defines and builds the container. Kubernetes handles the "where and how" — it decides which server to run it on, keeps it running, and scales it up or down based on demand.

4. The Kubernetes Architecture: What's Actually Running

Kubernetes operates as a cluster — a set of machines (called nodes) that it treats as a unified pool of compute.

┌─────────────────────────────────────────────────────────┐
│                    Control Plane                        │
│   API Server │ Scheduler │ Controller Manager │ etcd    │
└─────────────────────────────────────────────────────────┘
         │              │              │
    ┌────┴───┐      ┌───┴────┐    ┌───┴────┐
    │ Node 1 │      │ Node 2 │    │ Node 3 │
    │kubelet │      │kubelet │    │kubelet │
    │  Pod   │      │  Pod   │    │  Pod   │
    │  Pod   │      │  Pod   │    │        │
    └────────┘      └────────┘    └────────┘
  • Control Plane: The brain. Watches the cluster state, runs the scheduler, and reconciles "desired state" with "actual state."
  • Nodes: Worker machines — physical or virtual — that actually run your containers.
  • kubelet: An agent on every node that talks to the Control Plane and starts/stops containers as instructed.
  • etcd: A distributed key-value store that holds all cluster state. If Kubernetes is the brain, etcd is the long-term memory.

5. The Pod: Kubernetes' Atomic Unit

Docker's unit is the container. Kubernetes' unit is the Pod.

A Pod is a group of one or more containers that are always scheduled together on the same node. They share some resources but keep others isolated. This is where the namespace knowledge from the previous posts becomes directly applicable.

Which Namespaces Are Shared Inside a Pod?

When Kubernetes starts a Pod, it doesn't just launch your containers. It first starts a hidden container called the Pause container (also called the Infra container). This container does essentially nothing — it just holds namespaces open. Your actual application containers then join those namespaces.

Namespace Shared in a Pod? Effect
Network Yes All containers share one IP and port space. They reach each other via localhost.
IPC Yes Containers can use shared memory and message queues.
UTS Yes All containers in the Pod share the same hostname.
PID No (by default) Each container sees its own PID 1. Enable shareProcessNamespace: true to make the Pause container PID 1 and let containers signal each other's processes.
Mount (MNT) No Each container has its own isolated filesystem (UnionFS). Files are shared only via explicit Volumes.

Why the Pause Container?

In plain Docker, if you want two containers to share a network namespace, you start one, then start the second with --net=container:name. If the first container dies, the second loses its network too.

Kubernetes solves this with the Pause container. The network namespace is owned by the Pause container, not by any of your application containers. Your app containers come and go without affecting the namespace. The Pod's IP address stays stable for its entire lifetime.

Pod
├── Pause container  ←  owns the Network/IPC/UTS namespaces
├── Container A      ←  joins the Pause container's namespaces
└── Container B      ←  joins the Pause container's namespaces

Container A → localhost:5432 → Container B
(no networking hop, no bridge, no iptables — pure loopback)

The PID Namespace in Detail

By default, both containers in a Pod see their own process as PID 1 and cannot see each other's processes in ps or top. When you set shareProcessNamespace: true in the Pod spec:

  1. The Pause container becomes PID 1 for the entire Pod.
  2. All other containers become children of that Pause process.
  3. Container A can see Container B's processes and send signals to them — useful for debug sidecars that need to attach to a running process.

This is an opt-in because it breaks the isolation assumption most apps are written against. Use it only when you explicitly need cross-container process visibility.

The Sidecar Pattern

Shared namespaces inside a Pod enable a powerful architectural pattern: the sidecar. You run a small helper container alongside your main application in the same Pod. Common examples:

  • A log-shipping container that reads log files your app writes (via a shared Volume)
  • A metrics collector that scrapes your app's localhost:8080/metrics endpoint
  • An Envoy proxy that intercepts all network traffic for the app container

The sidecar has no idea where the app is — it's just localhost. The app has no idea the sidecar exists. They share a network and optionally a filesystem, but their process trees and images remain entirely separate.

How Containers in a Pod Communicate

There are three channels, each using a different shared resource:

  • Over the network: Use localhost:<PORT>. No bridge, no veth, no iptables — pure loopback. One constraint: two containers in the same Pod cannot bind to the same port.
  • Over the filesystem: Mount namespaces are not shared. Each container has its own UnionFS and cannot see the other's files by default. To share files, define a Kubernetes Volume at the Pod level and mount it into both containers. For example, an emptyDir volume mounted at /var/log in Container A and /data/logs in Container B gives both access to the same directory.
  • Over memory: IPC is shared. Containers can use SystemV shared memory or POSIX message queues to pass data without going through the network at all.

6. The Runtime: Where the Linux Primitives Actually Live

"Kubernetes containers are lighter" is often repeated but rarely explained. The claim is actually three separate claims bundled together, and they are not all equally true.

Claim 1: The isolation is lighter — False

At the kernel level, a container running in Kubernetes is identical to one running in Docker. The same namespaces, the same cgroups, the same UnionFS. There is no special lighter kernel magic for Kubernetes. 100MB of RAM in a Docker container is 100MB of RAM in a Kubernetes Pod.

Claim 2: The management daemon is lighter — True

The management path to start a container changed significantly.

The old path (via Docker daemon):

kubelet  →  Docker daemon  →  containerd  →  runC  →  Linux kernel

The Docker daemon is a full-featured suite: CLI, API server, build system, volume manager, image registry client. Kubernetes doesn't need any of that — it just needs to start a process. Running the Docker daemon on every cluster node was overhead for no benefit.

The modern path (via CRI):

kubelet  →  containerd  →  runC  →  Linux kernel

Kubernetes defines a Container Runtime Interface (CRI) — a minimal API for "start this container." containerd and CRI-O implement it directly. The tool that actually calls into the Linux kernel — runC — is the same in both paths. The container you get is byte-for-byte identical. What's different is the size of the management layer above it.

The practical difference: On a 500-node cluster, dropping the Docker daemon means 500 fewer heavy background processes consuming memory and CPU. That's real headroom returned to workloads.

Claim 3: Kubernetes images are lighter — Depends on how you build them

When people say Kubernetes containers are lightweight, they are often actually talking about how teams build images for production, not anything Kubernetes itself does.

  • Docker dev workflow: Use a full Ubuntu or Debian image with curl, vim, bash, and debugging tools installed. Convenient for interactive exploration.
  • Kubernetes prod workflow: Use Distroless images or statically compiled binaries. These images contain nothing but the compiled application — no shell, no package manager, no utilities. A Go binary that might normally live in a 400MB Ubuntu image ships in a 10MB distroless image instead.

Kubernetes makes this practical because it handles logging, metrics, and debugging externally — via sidecar containers, Prometheus exporters, and kubectl exec into ephemeral debug containers. Your main container doesn't need to include those tools.

7. What Kubernetes Gives You That Docker Cannot

Self-Healing

If a container crashes, Kubernetes restarts it automatically. If the node running it crashes, Kubernetes reschedules the Pod onto a healthy node. Docker Compose can restart a crashed container — but only on the same host, and only if that host is still up.

Horizontal Scaling

You can tell Kubernetes: "I want 10 copies of this Pod." It spreads them across nodes. When traffic drops, it scales back down. You can tie this to CPU or memory metrics with the Horizontal Pod Autoscaler — no manual intervention needed.

Rolling Updates with Zero Downtime

When you push a new version of your app, Kubernetes replaces Pods one at a time — bringing up a new Pod, waiting until it passes its health check, then terminating an old one. Your users never see downtime. Docker has no equivalent of this.

Service Discovery and Load Balancing

Kubernetes assigns a stable Service DNS name to a group of Pods. When you scale from 2 to 10 replicas, the Service automatically load-balances across all 10. The calling service doesn't know or care how many Pods are behind it.

Declarative State

You describe what you want (e.g., "3 replicas of this image, with these env vars, on nodes with at least 4GB RAM") in a YAML file. Kubernetes continuously reconciles reality against that description. If something drifts — a Pod dies, a node is replaced — Kubernetes fixes it without being asked.

8. When to Use What

Situation Right tool
Learning, local dev, testing a single image docker run
Multi-container app on one machine (dev or small prod) Docker Compose
Production app needing uptime, scaling, multi-node Kubernetes
Dozens of microservices with independent scaling needs Kubernetes

Kubernetes is not always the right answer. It is operationally complex and genuinely overkill for a small project or a team of one or two. Docker Compose handles a surprising amount of real-world load if your bottleneck is a single server. The jump to Kubernetes makes sense when you need multi-node resilience, automatic scaling, or zero-downtime deployments — and when you have the operational capacity to manage a cluster.

Key Takeaways

  • Docker and Kubernetes are not competitors — Docker builds containers, Kubernetes runs fleets of them across many machines.
  • Docker Compose is a single-host tool. It cannot spread load across servers, reschedule to healthy nodes, or auto-scale. That's the gap Kubernetes fills.
  • Kubernetes' atomic unit is the Pod, not the container. A Pod groups containers that must run together and share a Network, IPC, and UTS namespace via the Pause container.
  • Containers inside a Pod communicate over localhost — no bridge, no veth, no iptables involved.
  • The move from Docker daemon to CRI (containerd) didn't change how containers are isolated — namespaces and cgroups are identical. It removed the unnecessary overhead of the Docker daemon from every cluster node.
  • The Linux primitives from the previous posts — namespaces, cgroups, UnionFS — are the same at the bottom of every container, whether it's started by docker run, Docker Compose, or Kubernetes.
