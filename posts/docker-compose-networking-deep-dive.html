Docker Compose and Container Networking: How Linux Wires It All Together

The Starting Point

In the previous post, I covered what Docker actually is under the hood — namespaces for isolation, cgroups for resource limits, and Union File Systems for layered storage. One piece I left untouched was networking: how do containers actually talk to each other, and how does that story evolve from raw docker run all the way to Docker Compose?

Let's build up from first principles. Start with a concrete problem.

1. The Problem: Two Containers Need to Talk

Say you want to run a web app and a database. They live in separate containers. How do you get them to communicate?

The first thing you might try is just running both containers and hoping they can reach each other. And they sort of can — because Docker automatically attaches every container to something called the default bridge network.

The Default Bridge

When Docker is installed, it creates a network named bridge automatically. Every container you start with a bare docker run lands on this network. You can inspect it:

docker network inspect bridge

You'll see all your running containers listed in there, each with an IP like 172.17.0.2, 172.17.0.3, and so on. They can ping each other by IP. So the communication works, technically.

What Linux Actually Created

The default bridge is not a fiction — it's a real Linux network interface. Run ip addr on the host and you'll see docker0, a bridge interface sitting at 172.17.0.1. Each container gets a veth pair (a virtual ethernet cable): one end in the host namespace plugged into docker0, the other end renamed to eth0 inside the container's network namespace. Traffic flows through this bridge.

Drawbacks of the Default Bridge

No DNS. The default bridge does not provide name resolution. You cannot reach a container by its name — you must hardcode the IP address (172.17.0.3). But IPs are assigned dynamically. Every time a container restarts, it may get a different IP. Your app's config breaks.

No isolation. Every container on the host is on the same default bridge. A misconfigured app container can reach your database container, your cache, and anything else running on the same host — whether you intended that or not.

Not configurable. You cannot define custom subnets, MTU sizes, or any per-network settings on the default bridge.

So the default bridge gets you off the ground but falls apart the moment you need anything real. The next step is creating your own network.

2. The User-Defined Bridge — my-manual-net

Docker lets you create your own bridge network:

docker network create my-manual-net

Then start your containers explicitly on it:

docker run -d --name db --network my-manual-net postgres:15
docker run -d --name app --network my-manual-net my-web-app

Now your app container can reach the database with just db:5432 — no hardcoded IPs. Docker's embedded DNS server, which runs inside each user-defined network, resolves the container name to its current IP automatically.

What Linux Creates When You Run docker network create

When you run that command, Docker talks directly to the Linux kernel and sets up several real networking structures:

A. The Virtual Switch — the Bridge Interface

A new bridge interface appears on the host, named something like br-5f3a2c.... Run ip addr and you'll see it. This is a software Layer 2 switch — the hub that every container on this network "plugs into."

B. The Virtual Ethernet Pair — veth

When a container starts on my-manual-net, Docker creates a veth pair: a virtual patch cable with two ends. One end stays in the host's network namespace and gets plugged into br-5f3a2c.... The other end is injected into the container's private network namespace and renamed to eth0. Data written into one end comes out the other — like a real Ethernet cable.

C. IP Address Management — IPAM

Docker assigns a dedicated subnet to your network (e.g., 172.18.0.0/16) and acts as a DHCP server within it. Each container gets a unique IP from that pool. The bridge interface itself (172.18.0.1) is the Gateway.

D. NAT and Firewall Rules — iptables

Docker also programs your host's iptables. Two rules matter:

  • Masquerade (NAT): Outbound packets from containers are rewritten to look like they come from the host's physical IP, so they can reach the internet.
  • Isolation rules: Containers on my-manual-net are blocked from talking directly to containers on other user-defined networks. Each network is its own firewall zone.

How Containers Talk to Each Other on This Network

Container-to-container traffic inside the same network does not go through iptables. It stays on the bridge:

Container A (eth0: 172.18.0.2)
    │
    │  veth pair — virtual cable
    │
br-5f3a2c... (the Bridge — acts as a Layer 2 switch)
    │
    │  veth pair for Container B
    │
Container B (eth0: 172.18.0.3)

→  iptables is NOT involved. ARP + Bridge handles it entirely.

Container A sends an ARP broadcast asking "Who has 172.18.0.3?". The bridge knows which veth is connected to Container B and forwards the packet down that wire. Direct. Efficient. No kernel routing table involved.

iptables only enters the picture when a container needs to reach the internet:

Container A (172.18.0.2)  →  google.com
    │
    │  Bridge → Gateway (172.18.0.1)
    │
    │  iptables MASQUERADE: rewrite source IP to host's IP
    │
Host's WiFi / Ethernet  →  Internet

To see exactly what Docker built, run:

docker network inspect my-manual-net

The JSON output shows the exact Subnet, Gateway, and every container currently "plugged into" that virtual switch.

Drawbacks of the User-Defined Bridge

This is a significant improvement over the default bridge. But as soon as your stack grows beyond two containers, cracks start to show.

Everything is manual. You have to remember to create the network first, then pass --network my-manual-net to every single docker run command. Miss it once and a container ends up on the wrong network with no way to reach the others.

You manage startup order. If you start the app before the database is ready, the app crashes trying to connect. You have to script the ordering yourself.

Flags accumulate fast. A real container might need --name, --network, -p, -v, -e, --restart. A three-container stack turns into three long commands that you have to keep somewhere and remember to run in the right order.

Nothing is reproducible. If you lose your shell history, you don't know exactly how that running container was configured. Someone else on your team has to ask you. A README that documents the commands is just documentation debt.

Cleanup is tedious. To tear down the stack you have to stop each container, remove each container, and then remove the network — separately, in the right order.

3. Docker Compose — Solving Every Drawback

Docker Compose is the answer to all of the above. It is a declarative tool — instead of issuing imperative commands one by one, you describe the desired state of your entire stack in a single docker-compose.yml file, and Compose figures out how to make it happen.

The same two-container setup from above looks like this:

services:
  db:
    image: postgres:15
    environment:
      POSTGRES_PASSWORD: secret

  app:
    image: my-web-app
    ports:
      - "8080:80"
    depends_on:
      - db

Then:

docker compose up     # starts everything
docker compose down   # stops and removes everything, including the network

How Compose Solves Each Drawback

Drawback of manual approach What Compose does instead
Must manually create the network Compose creates a user-defined bridge for the project automatically
Must pass --network to every container All services in the file are joined to that network by default
Must manage startup order yourself depends_on declares ordering; Compose handles it
Long, forgettable docker run commands All config lives in the YAML file, committed to version control
Not portable — others must ask you Share the docker-compose.yml — that's the full spec
Teardown requires multiple manual steps docker compose down removes containers, networks, everything at once

The Isolation Is Identical

One thing worth stating clearly: Compose does not change how containers are isolated. Every container still gets its own namespaces and cgroups. The network Compose creates is still a user-defined bridge — the same Linux bridge, veth pairs, IPAM, and iptables rules described above. Compose is purely a management and orchestration layer on top of the same kernel primitives.

This Is a Software Defined Network on Your Laptop

Stepping back, what you end up with — whether through manual commands or Compose — is a full Software Defined Network (SDN) running on your machine. Your laptop's kernel is doing all of it.

Component Physical Equivalent Role
br-xxxx A Physical Switch Connects all containers together at Layer 2
veth pair An Ethernet Cable Plugs the container into the switch
iptables A Firewall / NAT Router Internet access and cross-network isolation
DNS (user-defined networks) A Phonebook Resolves service names like db to container IPs

Key Takeaways

  • The default bridge lets containers communicate by IP but gives you no DNS and no isolation — it breaks down immediately in any real scenario.
  • A user-defined bridge (docker network create) fixes both: you get DNS by container name and proper isolation. Under the hood, Docker creates a real Linux bridge interface, veth pairs, IPAM subnets, and iptables rules.
  • Container-to-container traffic on the same bridge is handled by ARP and the bridge itself — iptables is never involved.
  • iptables only comes into play for outbound internet traffic (NAT) and cross-network firewall rules.
  • The user-defined bridge still requires you to manually wire everything together with long docker run commands — which doesn't scale.
  • Docker Compose solves every operational drawback: it creates the user-defined network automatically, joins all services to it, handles startup ordering, and tears everything down with a single command.
  • Compose changes nothing about the underlying isolation — it's a management layer over the exact same Linux kernel primitives.
